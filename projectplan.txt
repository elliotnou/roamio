Roamio — Agentic Wellness Travel Companion
GenAI Genesis Hackathon | Sun Life Best Health Care Hack Using Agentic AI

Project Vision
Roamio is a mobile wellness companion for travellers that uses agentic AI to monitor energy levels throughout a trip and dynamically reroute the day's itinerary in real time. Unlike static travel apps, Roamio treats your physical and mental state as first-class data — building a personal fatigue profile over time and proactively adapting your plans before burnout happens.

Agentic AI Architecture
This is the core of the GenAI story. Gemini does not just answer a single prompt — it runs a multi-step agent loop with distinct responsibilities:

```
### Agent Step 1–2: Context Builder + Intent Classifier
```
System: You are a travel wellness AI. Given a traveller's current state, 
determine whether they need alternative activity suggestions.

User: {
  "energy_level": 4,
  "current_time": "2026-03-14T14:30:00",
  "current_activity": {
    "place_name": "Banff Upper Hot Springs",
    "activity_type": "hiking",
    "energy_cost_estimate": 8,
    "end_time": "2026-03-14T16:00:00"
  },
  "remaining_blocks_today": [...],
  "past_checkins_today": [{"energy_level": 6, "timestamp": "..."}]
}

Respond ONLY with valid JSON:
{
  "needs_rerouting": true,
  "energy_gap": 4,
  "reasoning": "one sentence"
}
```

### Agent Step 3: Place Resolver
```
System: You are a location extraction agent. Given a freeform activity 
description and destination city, return the most likely Google Places 
search query and activity classification.

User: {
  "place_name": "that big art museum downtown",
  "destination": "Toronto, Canada"
}

Respond ONLY with valid JSON:
{
  "google_places_query": "Art Gallery of Ontario Toronto",
  "activity_type": "museum",
  "energy_cost_estimate": 4
}
```

### Agent Step 4–5: Alternative Finder + Card Ranker
```
System: You are a travel rerouting agent. The traveller is low-energy.

than their current plan, suit their location, and can be completed in the 
remaining time block.

  "nearby_places": [...] // pre-fetched from Google Places API
}

Respond ONLY with valid JSON array:
[
  {
    "place_name": "Banff Legacy Trail Viewpoint",
    "place_id": "ChIJ...",
    "energy_cost_label": "low",
    "why_it_fits": "Flat paved path with mountain views — no elevation gain.",

  }

```
11. Dashboard detects active block based on current time
12. Check-in button → energy slider screen (emoji knob, 1–10)

### Phase 4 — Agentic Suggestion Engine
16. Google Places Nearby Search fires with current location + activity type
17. **Gemini Alternative Finder** receives Places results + context → returns ranked cards
18. Swipeable card UI (React Native Swiper or Reanimated gesture)
19. Card shows: place name, energy label, why-it-fits blurb, Google Maps deep link
20. User selection recorded in `check_ins.selected_suggestion_place_id`

### Phase 5 — Polish for Demo
21. Onboarding flow for first-time users
22. Empty states, loading skeletons, error boundaries
23. Smooth animations on energy slider and card swipe
24. PROGRESS.md and BLOCKERS.md up to date
```

Refined Scope Decisions
A few things from the original spec worth cleaning up before building:
Dropped for MVP:

Pattern/fatigue profile tracking — interesting long-term but requires multiple trips of data to be meaningful. A demo won't have this. Cut it.
Google Calendar ingestion — adds OAuth complexity with low demo payoff. Cut it.
Notifications — post-MVP only, irrelevant for web anyway.

Refined:

The "if energy is high, show an affirmation" path is fine for UX but shouldn't call the full agent pipeline. The intent classifier short-circuits immediately and returns a canned positive response — no wasted Gemini calls.
The Place Resolver running at itinerary-creation time (not check-in time) is the right call. Resolve once, store the result, use it everywhere. Don't re-resolve at check-in.
Energy slider emojis: keep them, they're good for the demo. But cap the emoji set at 5 distinct states mapped across the 1–10 range rather than 10 unique ones — cleaner visually.
Card swiper for suggestions: on web, implement as a horizontal scroll with keyboard/click navigation. The swipe gesture translates directly to React Native later with a library swap only.

Two Gemini calls total per check-in. Both return strict JSON. The intermediate Places API call is deterministic — Gemini only does reasoning, never hallucinating location data.
Why this is agentic and not just a chatbot: Each step has a defined role, typed inputs/outputs, and depends on the output of the prior step. The system makes decisions (reroute or affirm), calls external tools (Places API), then synthesizes results — all without user involvement after the initial slider input.

Gemini Agent Contracts
All calls use responseMimeType: "application/json" — no parsing fragility.
Call 1 — Intent Classifier
Purpose: Decide in one call whether rerouting is needed. Short-circuits the pipeline if energy is fine.
typescript// Input shape
interface ClassifierInput {
  energy_level: number;
  current_time: string;           // ISO
  current_block: {
    place_name: string;
    activity_type: ActivityType;
    energy_cost_estimate: number;
    end_time: string;
  };
  remaining_blocks_today: Array<{
    place_name: string;
    activity_type: ActivityType;
    start_time: string;
    end_time: string;
  }>;
  prior_checkins_this_trip: Array<{
    energy_level: number;
    timestamp: string;
  }>;
}

// Output shape (Gemini returns this exactly)
interface ClassifierOutput {
  needs_rerouting: boolean;
  energy_gap: number;             // how far below the activity's demand (0 if affirming)
  affirmation_message: string | null;   // populated if needs_rerouting = false
  reasoning: string;             // one sentence — shown in dev/debug panel
}
```
```
System prompt:
You are a travel wellness agent. A traveller has just reported their energy level
mid-trip. Given their current state and planned activities, decide if they need
alternative suggestions.

Reroute if: energy_level is more than 2 points below the current activity's
energy_cost_estimate, OR energy_level is 4 or below regardless.

If not rerouting, write a warm, specific affirmation_message referencing their
actual activity (e.g. "You're in great shape for the hot springs — enjoy it!").
If rerouting, set affirmation_message to null.

Return ONLY valid JSON matching the schema. No prose, no markdown.

Call 2 — Alternative Ranker
Purpose: Given pre-fetched Places candidates, reason over which ones best fit the user's energy state, time window, and activity style. Gemini does reasoning only — location data comes from the API.
typescript// Input shape
interface RankerInput {
  current_activity_type: ActivityType;
  energy_level: number;
  energy_gap: number;             // from classifier output
  time_remaining_minutes: number;
  destination: string;
  candidates: Array<{             // raw from Google Places Nearby Search
    place_id: string;
    name: string;
    types: string[];              // Google's place type array
    rating: number | null;
    user_ratings_total: number | null;
    distance_meters: number;
  }>;
}

// Output shape
interface RankerOutput {
  suggestions: Array<{
    place_id: string;             // must match a candidate's place_id exactly
    place_name: string;
    energy_cost_label: 'very low' | 'low' | 'moderate';
    why_it_fits: string;         // 1 sentence, specific and warm
    estimated_duration_minutes: number;
  }>;
}
```
```
System prompt:
You are a travel rerouting agent. The traveller is low on energy and needs
gentler alternatives to their current plan. You have a list of nearby places
from the Google Places API.

Select 3–5 candidates that:
- Are meaningfully less physically demanding than the original activity
- Can realistically be done in the time_remaining_minutes window
- Are varied in type (don't return 3 cafes)
- Have a rating of at least 3.5 if available

For each, write a why_it_fits sentence that is warm and specific — not generic.
Bad: "This is a relaxing option." Good: "A flat waterfront walk with great views,
no elevation and no time pressure."

IMPORTANT: Only use place_ids from the candidates list. Do not invent locations.
Return ONLY valid JSON. No prose, no markdown.

Build Order
Claude Code should complete each phase fully before starting the next. After each numbered step, run the dev server and confirm no console errors.
Phase 1 — Project Shell

Init Next.js 14 with TypeScript, Tailwind, App Router, shadcn/ui
Supabase client setup (lib/supabase.ts, lib/supabase-server.ts), env vars
Auth screens — login and signup wired to Supabase email auth
Auth guard in (app)/layout.tsx — redirect to login if no session
Basic nav shell with links to dashboard, trips, profile

Phase 2 — Trip & Itinerary

Trip list page reads from Supabase, shows empty state
Create trip form (destination, start/end date) → writes to trips
Activity block form (place name, start/end time) → writes to activity_blocks with resolved_* fields null initially
Wire api/resolve-place → calls Gemini Place Resolver → updates block row with resolved fields + activity_type + energy_cost_estimate
Day-grouped itinerary view under trip detail page

Phase 3 — Dashboard & Check-In

Dashboard reads active trip, computes active block from current time
"Check In" button → energy slider page with emoji states
api/checkin route — Step 1: runs Intent Classifier
If needs_rerouting = false: show affirmation screen, write check-in with agent_outcome: 'affirmed'

Phase 4 — Suggestion Engine

If rerouting needed: api/places-nearby fires with current GPS + activity_type
Results passed to Alternative Ranker (Gemini call 2)
Suggestion cards rendered in horizontal carousel
Each card: place name, energy label, why-it-fits blurb, Google Maps link
User selects or dismisses — check-in written with agent_outcome and selected_place_id

Phase 5 — Demo Polish

Loading states on all async operations (skeleton cards, spinner on check-in)
Empty states everywhere (no trips, no blocks, no suggestions found)
Mobile-responsive layout (the web app should already feel like a mobile app — single column, touch-friendly tap targets)
Error boundaries on API routes with user-facing messages
Seed script with a realistic demo trip so judges can see a populated state immediately

What Makes This Competitive for Judges

Genuine agentic pipeline — two specialized Gemini calls with typed contracts and an intermediate API call. The system decides, fetches, then synthesizes. That's an agent, not a chatbot.
Hallucination-resistant by design — Gemini never generates location data. It only reasons over Places API results. This is architecturally sound and worth explaining to judges explicitly.
Real health framing — energy as a wellness signal, overexertion prevention for anxious or first-time travellers, direct fit for the Sun Life track.
Clean separation for scalability — the agent logic in lib/agent/ is framework-agnostic by design, making the React Native migration story concrete and credible.
Demo-ready from the start — seed script means judges see a real populated trip, not a blank app.
